Les bases

Pour ceux qui voudraient un rappel des bases de Git, allez voir ici.

Une règle importante : On ne travaille jamais sur le master

Types de branche

Lorsqu’on veut contribuer au dépôt, on crée systématiquement une branche sur laquelle travailler. Une branche = une utilité (fonctionnalité, bug fix, ajout de ressources…)

Les noms de branche sont en kebab-case (minuscule sans accents avec des tirets).

On a 5 types de branches :

  • Les branches de fonctionnalité, pour développer les features principales du jeu. Une branche de fonctionnalité correspond en général à un ticket Jira.
    On nomme ces branches : feat/<ma-feature> comme par exemple feat/controller-vr

  • Les branches de bug fix, pour euh… fixer des bugs.
    On les nomme : fix/<mon-fix>, le nom de la branche peut-être une description du bug ou un numéro de ticket de bug Jira.

  • Les branches d’intégration, pour ajouter et intégrer des ressources au projet (modèles, anims, UI…)
    On les nomme : int/<mon-integration> par exemple, int/modele-peuple-1

  • Les branches de release qui servent de tronc pour préparer les release (mvp/alpha/beta/final). On y fait des bug fix et de l’intégration avant de release une version.
    On les nomme : release/<numero-de-version>

  • Les branches autres pour tout le reste. Vous pouvez les nommer comme vous voulez.

Pourquoi cette nomenclature ? D’abord, ça nous permet de mieux identifier les branches, mais aussi dans la plupart des GUI, les noms de branches construits comme des chemins chemin/vers-truc chemin/vers-machin s’affichent sous forme de “dossiers” donc c’est plus agréable pour les trier.

Le workflow

Voici le workflow que l’on va globalement suivre. Une branche = une utilité, on travaille dessus. On peut push cette branche si nécessaire, mais c’est mieux de la garder locale. Lorsque la feature fonctionne, on merge puis push le master.

Un pipeline Gitlab va ensuite tenter de build le jeu. Si le pipeline échoue, on peut retourner travailler sur la branche d’origine pour régler le soucis puis refaire un merge/push pour re-tester le pipeline.
